Cutting through the PQC noise — on my laptop

Real handshakes. Real numbers. Two migration scenarios tested on an HP Envy i7.

Kishore Chavali · Data Security Architect · Bangalore, India · April 2025

Why I built this lab

Over the last few months I have been immersed in PQC from every angle imaginable. NIST standards, NCSC and CISA migration guidance, crypto agility frameworks, position papers from standards bodies, vendor whitepapers, cryptographic posture management platforms, discovery and inventory tooling — I've read through most of it.

What struck me was the sheer range of signals coming in at once: genuine technical urgency sitting right next to significant marketing hype, real prioritisation problems buried under an avalanche of "harvest now, decrypt later" anxiety, and vendors pitching discovery tools before most organisations even have a clear view of what they're trying to discover.

Somewhere in the middle of all that, a few practical questions kept surfacing for me:

What actually needs to happen first?

What is the minimum viable migration step that builds momentum without requiring a full PKI overhaul?

And what do the NIST algorithms actually cost at the handshake level — on real hardware, not in a vendor benchmark?

Discovery and inventory are clearly foundational — you cannot migrate what you have not mapped. But I wanted to go a step further and understand the migration patterns themselves: what is genuinely easy to do today, what requires planning, and where the real operational cost lies.

So I set up a small lab on my personal HP Envy i7 laptop, built the Open Quantum Safe stack from source, generated both classical and post-quantum certificate chains, and ran two distinct TLS migration scenarios against each other. What follows are the actual results — commands, screenshots, handshake output, and benchmark numbers — with no vendor involvement and no synthetic environment.

This is a practitioner's PoC, not a production deployment guide. The goal is to build an informed intuition for the migration decision — and to give other architects something concrete to point at when the conversation turns to "but what does PQC actually cost?"

The lab stack

Everything runs on Ubuntu 24.04 (WSL2 on Windows 11) on the HP Envy i7 — no cloud, no dedicated server. The core components are the Open Quantum Safe project's liboqs 0.10.1 — which implements ML-KEM, ML-DSA, SLH-DSA and others from the NIST finalised standards — and oqs-provider 0.6.1, which bridges liboqs into OpenSSL's provider architecture so any tool using system OpenSSL automatically gains PQ algorithm support. Scenario B runs inside the openquantumsafe/oqs-ossl3 Docker image to work around an OpenSSL 3.0.x PEM encoder limitation for hybrid KEM key types.

BASH · Build stack 
# Build liboqs — core PQ algorithm library
git clone --depth 1 --branch 0.10.1 https://github.com/open-quantum-safe/liboqs.git
cmake -GNinja -B build -DOQS_USE_OPENSSL=ON -DBUILD_SHARED_LIBS=ON \
  -DOPENSSL_ROOT_DIR=/usr
ninja -C build && ninja -C build install

# Build oqs-provider — bridges liboqs into OpenSSL 3.x provider API
git clone --depth 1 --branch 0.6.1 https://github.com/open-quantum-safe/oqs-provider.git
cmake -GNinja -B build -Dliboqs_DIR=/usr/local/lib/cmake/liboqs
ninja -C build
sudo cp build/lib/oqsprovider.so /usr/lib/x86_64-linux-gnu/ossl-modules/

The two scenarios

I tested two distinct migration paths that reflect the actual decision enterprises face today.

Scenario A — KEM only
Replace key exchange with X25519+MLKEM768 hybrid (FIPS 203). Classical ECDSA P-256 certificate chain stays unchanged. Zero PKI impact.
ML-KEM-768 x25519 hybrid ECDSA cert
Scenario B — KEM + Auth
X25519+MLKEM768 key exchange AND ML-DSA-65 signed certificates (FIPS 204). Every part of the TLS handshake is quantum-safe.
ML-KEM-768 ML-DSA-65 PQ cert chain

Certificate size — the first surprise

Before running any handshake, I compared the cert sizes. This is the metric that will impact your PKI infrastructure most.

kishorechavali@Chavali — cert size comparison
Fig 2 — Scenario A cert (ECDSA P-256): 550 bytes. Scenario B cert (ML-DSA-65): 7,603 bytes. 13.8× increase.
Certificate Algorithm Size Ratio
Scenario A server cert ECDSA P-256 550 B baseline
Scenario B server cert ML-DSA-65 (FIPS 204) 7,603 B 13.8×
What this means operationally: ML-DSA-65 cert sizes will impact your CA infrastructure, OCSP response sizes, WAF and load balancer cert parsing buffers, and anything that stores or transmits certs at scale. Plan your PKI migration carefully — this is not a drop-in swap.

* Lab caveat — Scenario B: The 791ms figure includes Docker container startup overhead (~750ms) per handshake call in the benchmark harness. In a persistent production server where the container stays running, this overhead disappears entirely. The actual ML-DSA-65 signature verification cost is ~2–5ms. The real operational cost of Scenario B is the cert size, not the crypto speed.
Scenario A is the headline result: Hybrid KEM (X25519+MLKEM768) is not just acceptable — on loopback it outperforms classical X25519 due to lower latency variance. The migration cost is essentially zero. This is why Chrome, Cloudflare, and AWS are already shipping it in production.

Key takeaways for security architects

  • 01 Start Scenario A now. Hybrid KEM (X25519+MLKEM768) is already in Chrome, Cloudflare, and AWS. Near-zero performance cost. No PKI changes. The IETF hybrid design draft (0x11eb) is the emerging standard. There is no good reason to wait.
  • 02 Scenario B needs a PKI programme, not a config change. ML-DSA-65 cert sizes will cascade through your CA hierarchy, OCSP infrastructure, CRL distribution, WAF buffer configs, and any system that stores or transmits certs. Size up the blast radius before you migrate.
  • 03 "Harvest now, decrypt later" targets the KEM — not the signature. The quantum threat to confidentiality (long-lived secrets intercepted today) is active now. The threat to authentication has a longer horizon. Fix the KEM first — that is Scenario A.
  • 04 You can't migrate what you haven't inventoried. Build a Cryptographic Bill of Materials (CBOM) before you migrate. Know every TLS endpoint, every cipher suite, every cert algorithm, every HSM key type in your estate. Discovery is the hard part — migration is mechanical once you have the map.
